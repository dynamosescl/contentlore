UK FiveM Frameworks.

The three frameworks UK FiveM servers actually run on — ESX, QBCore, QBox. What each does, what it costs in performance, and what choosing one in 2026 reveals about when a server was conceived. Maintained as a permanent reference.

Last updated · 19 April 2026 Review cadence · Quarterly Scope · ESX / QBCore / QBox

What a framework is

A FiveM framework is the software layer that sits between FiveM's platform and a server's game logic. It handles player data, inventory, jobs, vehicles, money, permissions, and the other primitives every roleplay server needs. A server operator does not write these systems from scratch; they pick a framework and build on top of it. The framework choice is, for that reason, one of the most consequential technical decisions made about a server.

Three frameworks dominate the UK and global FiveM roleplay scene in 2026. ESX is the oldest and most-used by raw server count. QBCore is the historical standard for roleplay servers, with an enormous script ecosystem. QBox is a modern fork of QBCore, built by former QBCore contributors, that is growing quickly. Each framework represents a different philosophical and technical approach to the same problem space.

The exact numerical share between the three has fluctuated through 2026 and is not public in audited form. The ordering, however, is stable: ESX first, QBCore second, QBox third but closing.

ESX

ESX — full name EssentialMode Extended — is the longest-established FiveM roleplay framework in active wide use. It emerged as a community successor to the original EssentialMode and quickly became the default choice for economy-based roleplay servers. Its mature ecosystem covers virtually every possible RP script requirement; its multi-language community support is strong; its performance characteristics are measurably better than either QBCore or QBox.

Where ESX is weaker is in modern coding conventions. The framework predates many patterns that newer FiveM resources assume, which can create friction when integrating recent scripts written against QBCore or QBox conventions. For servers with mature custom codebases built on ESX, this rarely matters. For new servers being built in 2026, ESX is a choice that trades modern ergonomics for battle-tested stability and performance.

Editorial observation

ESX is the framework of the server that has been running for five years, has its own script library, and is not rewriting its foundation for anybody. It is quietly the best choice for servers with professional engineering discipline, and quietly the worst choice for servers trying to keep up with the Cfx forum's monthly releases.

When to pick ESX

  • The server prioritises performance and scales to high player counts.
  • The operator has existing ESX codebase or experience with the conventions.
  • The server operates as a mature, economy-heavy roleplay environment.
  • Script stability matters more than access to the latest community releases.

QBCore

QBCore rose to prominence from around 2020 as a more beginner-friendly alternative to ESX. Its explicit philosophy was accessibility: clear documentation, structured starting points, a collection of bundled scripts that gave new server operators a complete working RP server out of the box. That accessibility built the largest script ecosystem in FiveM — more community-contributed resources target QBCore than any other framework.

That ecosystem is also QBCore's central problem in 2026. The framework's codebase is several years old; its performance costs are the highest of the three frameworks covered here; and its active development has slowed since around 2023. Many experienced server operators who built on QBCore in its heyday have either migrated to QBox or are considering doing so. The community sentiment has shifted from "QBCore is the obvious starting point" to "QBCore is the framework you are inheriting, not the framework you are choosing."

Editorial observation

QBCore is the framework of the server that was built in 2021 or 2022 and has grown faster than its foundations can comfortably support. The framework is not broken; it is ageing. What that means for an operator depends on whether the server's growth trajectory is exposing QBCore's limits, or whether stability within those limits is acceptable. Both positions are defensible.

When to pick QBCore

  • The server has an established QBCore codebase and no compelling reason to migrate.
  • The operator is prioritising the largest possible library of community scripts.
  • Performance headroom is adequate for the server's current and projected player count.
  • The operator prefers QBCore's conventions to QBox's stricter modern patterns.

QBox

QBox was created on 27 September 2022 as a fork of QBCore by a team of former QBCore contributors. Its explicit philosophy is modernisation — performance, security, clean code, and deep integration with the ox resource ecosystem (ox_lib, ox_inventory, oxmysql) that has become the highest-quality tooling layer in FiveM. QBox ships with ox_inventory by default, which eliminates one of the most common and damaging mistakes new server operators make: starting with a poor inventory system that is expensive to replace later.

The strategic move QBox made, and the reason it is growing faster than either alternative, is backwards compatibility with QBCore. QBox can run most QBCore scripts with minimal or no modification. That means a QBCore server owner facing performance problems can migrate to QBox without giving up the script ecosystem they spent years building. Community research in late 2025 suggested a meaningful share of QBCore server owners were either in the process of switching to QBox or wished they had started on it. The direction of travel is clear.

Editorial observation

QBox is the framework of the server that is being built or rebuilt in 2025 or 2026. If a UK server announces a framework migration over the next twelve months, the destination is QBox more often than not. The combination of modern architecture, backwards compatibility, and active development makes QBox the lowest-friction choice for operators who want to modernise without starting over.

When to pick QBox

  • The server is being built fresh in 2025 or 2026.
  • The operator is migrating from QBCore because of performance or security concerns.
  • Access to the ox ecosystem of modern resources is a priority.
  • The operator wants active development and modern conventions over legacy stability.

What framework choice reveals

For the scene-watching reader rather than the server operator, framework choice is also a signal about when and how a server was conceived. A UK server running ESX in 2026 is telling you its core architecture probably predates 2021 and its engineering team values stability over novelty. A server on QBCore is most likely a 2021-to-2023 build that has not migrated. A server on QBox is either a 2023+ build or a QBCore server that has made the modernisation jump.

This reading is not definitive — operators choose frameworks for reasons that are not always visible from outside — but it is consistently useful. When this desk's Scoreboard coverage describes a server, the framework running underneath is one of the quieter data points about its technical maturity. Servers that have kept their framework choice under review and migrated when appropriate tend to be the ones that scale cleanly. Servers that have stayed on the same framework for five years without revisiting the choice tend to be the ones whose technical debt shows.

What Enhanced does to framework choice

Cfx has stated that existing FiveM scripts — written in Lua, JavaScript, or C# — remain backwards-compatible on FiveM for GTAV Enhanced. That means, in principle, ESX, QBCore, and QBox servers can all migrate to Enhanced without framework replacement. The migration work is at the platform layer, not the framework layer.

In practice, the Enhanced migration creates a natural inflection point where operators reconsider framework choice anyway. A server doing the work to move to Enhanced is already rebuilding parts of its stack; adding a framework migration to that work is less costly than doing it separately. This desk's expectation is that Enhanced migration will accelerate framework churn — specifically, a wave of QBCore-to-QBox migrations bundled with the Legacy-to-Enhanced move. Whether that expectation holds up will be reviewable in this desk's Q3 and Q4 coverage.

What this page does not cover

Three scope exclusions worth stating explicitly.

Custom forks and private frameworks. Some of the largest UK servers run custom frameworks — either forks of ESX, QBCore, or QBox with substantial private modifications, or fully bespoke systems. This reference covers the public, widely-adopted frameworks; private systems are out of scope because they are, by definition, not community-available.

Non-RP frameworks. This page focuses on roleplay frameworks because UK FiveM is overwhelmingly a roleplay scene. FiveM supports deathmatch, racing, and other non-RP game modes, which have different framework ecosystems. Those are out of scope here.

Framework configuration details. This page describes what each framework is and when to choose it. It does not cover installation, configuration, or script development. The official docs for each framework do that work and are linked in the sources block.
